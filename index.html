Chapter 01
Why Playwright & TypeScript

Playwright is a modern end-to-end testing framework developed by Microsoft that enables reliable, fast, and cross-browser web testing. Combined with TypeScript's static type system, it creates a powerful foundation for test automation that catches bugs before they reach production.

The Modern Testing Landscape

The shift from manual QA to automated testing demands tools that are not just functional but resilient. Flaky tests erode trust in your test suite. Playwright was built from the ground up to eliminate the root causes of test flakiness: race conditions, timing issues, and brittle selectors.

Figure 1.1 — Playwright Multi-Browser Architecture

Key Advantages

Auto-Waiting

Automatically waits for elements to be visible, stable, and actionable before interacting.

Zero Flakiness

Built-in retry, network idle detection, and actionability checks eliminate race conditions.

Cross-Browser

Single API for Chromium, Firefox, and WebKit with engine-level control.

TypeScript Native

First-class TypeScript support with full type definitions and IntelliSense.

Parallel Execution

Run tests across multiple workers, browsers, and shards simultaneously.

Trace Viewer

Rich post-mortem traces with DOM snapshots, network, and console logs.

Playwright vs. Alternatives

Auto-WaitFull actionability checksPartialManualManual

Tip: Start with one stable browser project first (usually Chromium), make tests reliable, then expand to Firefox/WebKit for true cross-browser confidence.

# Verify installation

Recommended Project Structure

Structure Review for Playwright 1.50+

  • Current structure is solid for a starter suite: clear separation of specs, fixtures, and pages.
  • Add a dedicated auth setup flow (for example tests/setup/auth.setup.ts) and store reusable auth state under playwright/.auth/.

Chapter 03
Locators & Selectors Deep Dive

Playwright's locator system is the backbone of reliable test interaction. Unlike CSS selectors or XPath, Playwright's locators are resilient to DOM structure changes and align with how users perceive the page.

Locator Priority Order

Figure 3.1 — Locator priority pyramid (1.50+ best practices)
Philosophy: While getByTestId() is highly resilient to DOM changes, Playwright prioritizes user-facing locators (role, text, label, etc.) that reflect how users and assistive technology interact with your app. Use test IDs only when you can't locate by user-facing attributes. This ensures tests align with actual user experience rather than implementation details.

Locator Methods

MethodUse CaseExample
getByRole()Buttons, links, form elements by ARIA rolepage.getByRole('button', { name: 'Submit' })
getByText()Find by visible text contentpage.getByText('Welcome back')
getByLabel()Form inputs associated with a labelpage.getByLabel('Email address')
getByPlaceholder()Inputs with placeholder textpage.getByPlaceholder('Enter name')
getByTestId()Elements with data-testid attributepage.getByTestId('submit-btn')
getByAltText()Images and media by alt textpage.getByAltText('Company logo')
getByTitle()Elements with title attributepage.getByTitle('Close dialog')
locator()CSS or XPath selector (fallback)page.locator('.card > .title')

Chaining & Filtering

locators-chain.ts 
// Chain locators to narrow down to specific elements
const row = page.getByRole('row', { name: 'John Doe' });
await row.getByRole('button', { name: 'Delete' }).click();

// Filter by child text content
const items = page.getByRole('listitem').filter({ hasText: 'In Stock' });

// Filter by having a child element
const cards = page.locator('.card').filter({
  has: page.getByRole('img')
});

// Nth element (0-indexed)
const firstItem = page.getByRole('listitem').nth(0);
const lastItem = page.getByRole('listitem').last();
Avoid .nth() when possible. It couples your test to DOM order. Prefer filtering by text content or using data-testid for unique identification.
Tip: Configure a custom test id attribute once with testIdAttribute in config so locator strategy stays consistent across teams and apps.

Chapter 04
Interactions & User Simulations

Core Interactions

Every interaction in Playwright automatically performs actionability checks: the element must be visible, stable, receive events, and not be obscured by other elements.

interactions.spec.ts 
import { test, expect } from '@playwright/test';

test('form submission flow', async ({ page }) => {
  await page.goto('/contact');

  // Text input
  await page.getByLabel('Name').fill('Jane Smith');

  // Clear and enter text (locator.type is deprecated in modern Playwright)
  await page.getByLabel('Email').clear();
  await page.getByLabel('Email').fill('jane@example.com');

  // Select dropdown option
  await page.getByLabel('Category').selectOption('support');

  // Checkbox and radio
  await page.getByLabel('Accept terms').check();
  await page.getByLabel('Priority: High').check();

  // File upload
  await page.getByLabel('Attachment').setInputFiles('./fixtures/doc.pdf');

  // Submit
  await page.getByRole('button', { name: 'Send' }).click();

  // Verify success
  await expect(page.getByText('Message sent!')).toBeVisible();
});

Advanced Mouse & Keyboard

advanced-interactions.ts 
// Drag and drop
await page.locator('#source').dragTo(page.locator('#target'));

// Hover and click nested menu
await page.getByRole('link', { name: 'Products' }).hover();
await page.getByRole('link', { name: 'Electronics' }).click();

// Keyboard shortcuts
await page.keyboard.press('Control+a');
await page.keyboard.press('Control+c');
await page.keyboard.insertText('Pasted content');

// Multi-select with Shift+Click
await item1.click();
await page.keyboard.down('Shift');
await item5.click();
await page.keyboard.up('Shift');

Interaction Lifecycle

Figure 4.1 — Actionability checks performed before every interaction
Force option: Use .click({ force: true }) only as a last resort. It bypasses all actionability checks and can lead to hidden flakiness.
Tip: Prefer user-intent APIs such as fill, selectOption, and role-based clicks over low-level mouse coordinates for better stability.

Chapter 05
Assertions & Verification

Playwright includes a rich assertion library built on top of expect. Unlike simple assertions, Playwright's assertions automatically retry until the condition is met or a timeout is reached, eliminating timing-related flakiness.

Auto-Retrying Assertions

assertions.spec.ts 
import { test, expect } from '@playwright/test';

test('dashboard loads with data', async ({ page }) => {
  await page.goto('/dashboard');

  // Visibility assertions (auto-retry)
  await expect(page.getByText('Revenue')).toBeVisible();
  await expect(page.getByText('Loading...')).toBeHidden();

  // Text content assertions
  await expect(page.locator('.total')).toHaveText('$12,450');
  await expect(page.locator('.total')).toContainText('$12');

  // Attribute & class assertions
  await expect(page.getByRole('button')).toHaveAttribute('disabled');
  await expect(page.locator('.alert')).toHaveClass(/error/);

  // Count assertions
  await expect(page.getByRole('row')).toHaveCount(25);

  // State assertions
  await expect(page.getByLabel('Email')).toBeEnabled();
  await expect(page.getByLabel('Email')).toBeFocused();
  await expect(page.getByLabel('Terms')).toBeChecked();
});

Page-Level Assertions

page-assertions.ts 
// URL assertions
await expect(page).toHaveURL('/dashboard');
await expect(page).toHaveURL(/\/dashboard\/\d+/);

// Title assertion
await expect(page).toHaveTitle('Dashboard - My App');

// Screenshot comparison
await expect(page).toHaveScreenshot('dashboard.png');

// Custom timeout for slow operations
await expect(page.getByText('Processed')).toBeVisible({ timeout: 30_000 });

Soft Assertions

soft-assertions.ts 
// Soft assertions collect all failures instead of failing on first
test('validate all dashboard cards', async ({ page }) => {
  await page.goto('/dashboard');

  await expect.soft(page.getByText('Revenue')).toBeVisible();
  await expect.soft(page.getByText('Users')).toBeVisible();
  await expect.soft(page.getByText('Orders')).toBeVisible();
  // Test continues even if some assertions fail
  // All failures reported at the end
});
Prefer auto-retrying assertions over manual waits. Instead of await page.waitForSelector() + expect(...).toBeTruthy(), use await expect(locator).toBeVisible() directly.

Chapter 06
Page Objects & Test Composition

The Page Object Model (POM) encapsulates page structure and interactions into reusable classes. Modern Playwright favors composition over inheritance, integrating page objects with fixtures for flexible test orchestration.

Modern POM Pattern (Composition-Based)

Figure 6.1 — Modern POM with composition (Playwright 1.50+)
Composition vs Inheritance: Modern Playwright recommends flat page objects that receive the page fixture via constructor injection, avoiding complex inheritance hierarchies. Each page object is independent and composed into tests via fixtures. This provides better flexibility, testability, and aligns with Playwright's fixture-based architecture. The traditional BasePage inheritance pattern (common in Selenium) is still valid but less idiomatic in Playwright's ecosystem.

Modern LoginPage (Composition Pattern)

pages/LoginPage.ts 
import { Page, Locator } from '@playwright/test';

export class LoginPage {
  readonly page: Page;
  readonly emailInput: Locator;
  readonly passwordInput: Locator;
  readonly submitButton: Locator;
  readonly errorMessage: Locator;

  constructor(page: Page) {
    this.page = page;
    this.emailInput = page.getByLabel('Email');
    this.passwordInput = page.getByLabel('Password');
    this.submitButton = page.getByRole('button', { name: 'Sign in' });
    this.errorMessage = page.getByRole('alert');
  }

  async goto() {
    await this.page.goto('/login');
  }

  async login(email: string, password: string) {
    await this.emailInput.fill(email);
    await this.passwordInput.fill(password);
    await this.submitButton.click();
  }
}

Traditional Pattern (BasePage Inheritance)

While modern Playwright recommends composition, some teams (especially migrating from Selenium) use inheritance:

pages/BasePage.ts 
import { Page, Locator } from '@playwright/test';

export class BasePage {
  readonly page: Page;
  readonly header: Locator;

  constructor(page: Page) {
    this.page = page;
    this.header = page.getByRole('banner');
  }

  async navigate(path: string) {
    await this.page.goto(path);
  }
}
pages/LoginPage.ts (with inheritance) 
import { Page, Locator } from '@playwright/test';
import { BasePage } from './BasePage';

export class LoginPage extends BasePage {
  readonly emailInput: Locator;
  readonly passwordInput: Locator;
  readonly submitButton: Locator;

  constructor(page: Page) {
    super(page);
    this.emailInput = page.getByLabel('Email');
    this.passwordInput = page.getByLabel('Password');
    this.submitButton = page.getByRole('button', { name: 'Sign in' });
  }

  async login(email: string, password: string) {
    await this.emailInput.fill(email);
    await this.passwordInput.fill(password);
    await this.submitButton.click();
  }
}
Anti-Pattern: Putting assertions inside Page Objects. POM classes should only encapsulate actions and state queries. Assertions belong in test specs for clear separation of concerns.

Page Object Model vs Page Fixtures

AspectPage Object Model (POM)Page Fixtures
Primary purposeEncapsulate page-level UI behavior into reusable classes.Provide test dependencies (pages, data, auth, APIs) through Playwright's fixture lifecycle.
Best forStable business flows used in many tests (login, checkout, admin actions).Cross-cutting setup/teardown, shared state, environment wiring, and test composition.
MaintainabilityHigh when page methods are cohesive and domain-oriented.High for test orchestration, can become complex if over-nested.
Performance controlNeutral by itself.Strong: worker-scoped fixtures, lazy creation, and reuse reduce suite time.
Common pitfallTurning classes into mini test frameworks with assertions and branching logic.Putting too much UI behavior inside fixtures instead of dedicated page objects.

Which Is Better?

For most real-world test suites, the best approach is hybrid: use POM for UI behavior and fixtures for orchestration. If you must pick one, fixtures usually provide more architectural leverage because they manage lifecycle, dependency injection, and scaling across projects.

Recommendation: Keep page interactions in classes (POM) and compose those classes via fixtures. This gives clean APIs, faster suites, and less duplicated setup code.
hybrid-pattern.spec.ts 
import { test as base, expect } from '@playwright/test';
import { LoginPage } from '../pages/LoginPage';

type Fixtures = {
  loginPage: LoginPage;
};

const test = base.extend<Fixtures>({
  loginPage: async ({ page }, use) => {
    await use(new LoginPage(page));
  },
});

test('user can sign in', async ({ loginPage }) => {
  await loginPage.navigate('/login');
  await loginPage.login('user@test.com', 'secret');
  await expect(loginPage.page).toHaveURL(/dashboard/);
});
Tip: Keep Page Objects focused on reusable business actions, and let fixtures manage lifecycle concerns like auth, test data bootstrapping, and cleanup.

Chapter 07
Handling Async & Waiting

Playwright's auto-waiting mechanism handles most timing scenarios automatically. However, understanding the waiting strategies is essential for complex async flows like polling, websockets, and long-running operations.

Wait Strategies Comparison

StrategyWhen to UseRisk Level
Auto-wait (default)Element visibility, stability before actions — built-in to all interactionsNone
expect(locator).toBeVisible()Explicit assertions with auto-retry for dynamic contentLow
page.waitForLoadState()Wait for page lifecycle events (load, domcontentloaded, networkidle)Low
page.waitForURL()Wait for navigation to specific URL or patternLow
page.waitForResponse()Wait for specific API responses during user actionsMedium
page.waitForFunction()Custom JavaScript conditions (use sparingly)Medium
page.waitForTimeout()Debugging only — never in production testsHigh

Waiting for Network Responses

network-wait.ts 
test('wait for API then verify UI', async ({ page }) => {
  // Start waiting for response before triggering action
  const responsePromise = page.waitForResponse('**/api/users');
  await page.getByRole('button', { name: 'Load Users' }).click();
  const response = await responsePromise;

  // Verify the response data
  const body = await response.json();
  expect(body.users).toHaveLength(10);

  // Wait for response with specific status
  const createPromise = page.waitForResponse(
    resp => resp.url().includes('/api/orders')
      && resp.status() === 201
  );
  await page.getByRole('button', { name: 'Place Order' }).click();
  await createPromise;
});

// Wait for multiple requests/responses together
test('wait for parallel API calls', async ({ page }) => {
  const [usersResp, settingsResp] = await Promise.all([
    page.waitForResponse('**/api/users'),
    page.waitForResponse('**/api/settings'),
    page.goto('/dashboard')
  ]);
  expect(usersResp.ok()).toBe(true);
});

Navigation & Load State Waiting

load-state.ts 
// Wait for specific load states
await page.goto('/dashboard');
await page.waitForLoadState('networkidle'); // All network connections idle

// Wait for URL to match pattern
await page.getByRole('link', { name: 'Settings' }).click();
await page.waitForURL('**/settings');
await expect(page).toHaveURL(/\/settings$/);

// Wait for DOM to be ready (faster than 'load')
await page.goto('/heavy-page');
await page.waitForLoadState('domcontentloaded');
await expect(page.getByRole('heading')).toBeVisible();

Custom Conditions & Polling

custom-waiting.ts 
// Prefer locator assertions over waitForFunction
const tasks = page.locator('.task-item');
await expect(tasks).toHaveCount(5, { timeout: 10000 });

// Wait for custom JavaScript condition (use sparingly)
await page.waitForFunction(
  () => (window as any).appState === 'initialized',
  { timeout: 5000 }
);

// Wait for element count to change dynamically
await page.waitForFunction(
  (minCount) => document.querySelectorAll('.notification').length >= minCount,
  3 // Pass argument to function
);

// Wait for WebSocket connection
const wsPromise = page.waitForEvent('websocket', {
  predicate: ws => ws.url().includes('/ws/updates')
});
await page.getByRole('button', { name: 'Connect' }).click();
const ws = await wsPromise;

Timeout Configuration (Playwright 1.50+)

timeout-config.ts 
// Per-assertion timeout
await expect(page.getByText('Loading...')).toBeHidden({ timeout: 15000 });

// Per-action timeout
await page.getByRole('button', { name: 'Submit' }).click({ timeout: 10000 });

// Set default timeout for test
test.setTimeout(60000); // 60 seconds for entire test

// Configure action and navigation timeouts in config
export default defineConfig({
  use: {
    actionTimeout: 10000,      // Default for clicks, fills, etc.
    navigationTimeout: 30000, // Default for page.goto()
  }
});
Never use waitForTimeout() in production tests. Hard-coded waits make tests slow and flaky. If the app responds faster, you waste time. If slower, the test fails. Always use condition-based waiting.
Tip: Wait for user-visible outcomes (UI state, response predicates, or URL/title assertions) instead of implementation timing details.

Network Interception & Mocking

Network interception allows you to intercept, modify, or block network requests. This is invaluable for testing edge cases, error states, and isolating tests from external dependencies.

Interception Flow

Figure 8.1 — Network interception & mock flow

Mocking API Responses

mocking.spec.ts 
import { test, expect } from '@playwright/test';
import mockUsers from '../test-data/users.json';

test('displays mocked user data', async ({ page }) => {
  // Intercept and return mock data
  await page.route('**/api/users', async route => {
    await route.fulfill({
      status: 200,
      contentType: 'application/json',
      body: JSON.stringify(mockUsers),
    });
  });

  await page.goto('/users');
  await expect(page.getByRole('row')).toHaveCount(mockUsers.length);
});

test('handles API error gracefully', async ({ page }) => {
  await page.route('**/api/users', route =>
    route.fulfill({ status: 500, body: 'Server Error' })
  );

  await page.goto('/users');
  await expect(page.getByText('Failed to load')).toBeVisible();
});

test('blocks analytics tracking', async ({ page }) => {
  await page.route('**/analytics/**', route => route.abort());
  await page.goto('/');
  // No analytics calls made
});
Har files: Use page.routeFromHAR('recording.har', { update: false }) to replay recorded network traffic. Record once with --record-har flag, then replay in tests for fast, deterministic execution.

Chapter 09
Visual Regression Testing

Visual regression testing captures screenshots and compares them against approved baselines. Playwright's built-in screenshot comparison engine handles anti-aliasing differences, color spaces, and rendering variations across browsers.

Screenshot Comparison Types

1

Full Page Screenshot

Captures the entire scrollable page including below-the-fold content.

2

Element Screenshot

Captures a specific component or section, ignoring the rest of the page.

3

Viewport Screenshot

Captures only the visible viewport area — good for responsive tests.

4

Component Isolated

Uses toHaveScreenshot() on a specific locator for component-level checks.

Visual Testing in Practice

visual.spec.ts 
test('homepage visual regression', async ({ page }) => {
  await page.goto('/');

  // Full page — generates __screenshots__/homepage-full.png
  await expect(page).toHaveScreenshot('homepage-full.png', {
    fullPage: true,
    animations: 'disabled',
  });

  // Component-level — only the hero section
  await expect(page.locator('.hero-section')).toHaveScreenshot('hero.png', {
    maxDiffPixelRatio: 0.01,  // Allow 1% pixel difference
  });
});

test('dark mode comparison', async ({ page }) => {
  await page.goto('/');
  await page.emulateMedia({ colorScheme: 'dark' });
  await expect(page).toHaveScreenshot('homepage-dark.png');
});

Configuration Options

OptionDescriptionDefault
thresholdAllowed perceived color difference (0-1)0.2
maxDiffPixelRatioMax ratio of differing pixels0
maxDiffPixelsMax absolute number of differing pixels0
animations"disabled" | "allow""allow"
caret"hide" | "initial""initial"
scale"css" | "device""device"
Updating baselines: Use npx playwright test --update-snapshots to update all baselines. Review every diff carefully — never blindly update without understanding what changed.
Tip: Mask dynamic regions (dates, ads, rotating banners) before screenshot assertions to reduce false-positive visual diffs.

CI/CD Pipeline Integration

Running Playwright tests in CI requires careful configuration for browser installation, caching, parallel execution, and artifact collection. Here's how to set up a production-grade pipeline.

CI Pipeline Architecture

Figure 10.1 — CI/CD pipeline stages

GitHub Actions Configuration

.github/workflows/playwright.yml 
name: Playwright Tests
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      fail-fast: false
      matrix:
        shard: [1/4, 2/4, 3/4, 4/4]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      - name: Install dependencies
        run: npm ci
      - name: Install Playwright browsers
        run: npx playwright install --with-deps
      - name: Run Playwright tests
        run: npx playwright test --shard=${{ matrix.shard }}
      - uses: actions/upload-artifact@v4
        if: ${{ failure() }}
        with:
          name: test-results-${{ matrix.shard }}
          path: test-results/
          retention-days: 7
Sharding: Split tests across 4 parallel jobs with --shard=N/4. Each shard runs a subset of tests. Total wall-clock time drops to roughly 1/4 of single-job execution.

Docker-Based Execution

Use official Microsoft Playwright Docker images for consistent environments across local and CI execution. No browser installation needed — images come pre-bundled.

Chapter 11
Best Practices & Anti-Patterns

Do's and Don'ts

Do This
  • Use role-based and text-based locators
  • Use test.describe blocks for grouping
  • Use custom fixtures for shared setup
  • Set baseURL in config, not in each test
  • Use test.slow() for known slow tests
  • Tag tests with @smoke, @regression
  • Use storage state for auth persistence
  • Keep tests independent and order-free
Avoid This
  • CSS selectors like .btn > span
  • XPath for element selection
  • waitForTimeout() in production
  • Hard-coded sleep/delay values
  • Tests that depend on run order
  • Sharing mutable state between tests
  • Testing implementation details
  • Ignoring flaky tests instead of fixing

Custom Fixtures Pattern

tests/fixtures/auth.fixture.ts 
import { test as base, Page } from '@playwright/test';
import { LoginPage } from '../../pages/LoginPage';

type AuthFixtures = {
  authenticatedPage: Page;
  loginPage: LoginPage;
};

export const test = base.extend<AuthFixtures>({
  loginPage: async ({ page }, use) => {
    await use(new LoginPage(page));
  },
  authenticatedPage: async ({ page }, use) => {
    // Reuse stored authentication state

  },

Playwright 1.50+ and 1.60 Highlights

FeatureVersionWhy It Matters
test.step(..., { timeout })1.50Set timeout per step, not only per test.
test.step.skip()1.50Skip individual steps while keeping test flow readable.
